# 產品需求文件 (PRD)：個人記帳簿系統

## 1. 專案概述
- **背景與動機**：現代人生活繁忙，收支項目繁雜，若沒有良好的記帳習慣，容易造成財務管理上的困難。為了幫助使用者更清晰地掌握個人財務狀況，我們需要一個簡單易用、能隨時記錄與查詢的個人記帳網站。
- **目標用戶**：
  - **一般人**：希望能夠快速記帳、追蹤日常開銷與收入的普通使用者。
  - **管理員**：負責維護系統運作、管理使用者狀態與系統全域設定的系統管理者。
- **核心價值主張**：提供一個輕量、直覺的記帳平台，讓使用者能輕鬆記錄每一筆收支，並透過報表與搜尋功能，隨時掌握財務流向，達成更好的個人理財目標。

## 2. 功能需求

### 主要功能清單
1. **記帳功能（新增收支）**
2. **瀏覽與搜尋紀錄**
3. **更改與刪除內容**
4. **收支統計與圖表**
5. **分類管理**
6. **後台使用者管理（管理員專屬）**

### 使用者故事 (User Stories)
- **記帳功能**：作為「一般人」，我希望「能快速新增一筆收支紀錄（包含金額、分類、日期、備註）」，以便「隨時記錄我的花費與收入」。
- **瀏覽與搜尋**：作為「一般人」，我希望「能用關鍵字或日期範圍搜尋過去的記帳紀錄」，以便「快速找到某筆特定的開銷」。
- **更改內容**：作為「一般人」，我希望「能編輯或刪除已經新增的記帳紀錄」，以便「在輸入錯誤時進行修正」。
- **收支統計**：作為「一般人」，我希望「能看到每月的收支總計摘要」，以便「了解我的主要花費項目」。
- **分類管理**：作為「一般人」，我希望「能自訂記帳分類（如：餐飲、交通、娛樂）」，以便「更彈性地管理我的收支項目」。
- **使用者管理**：作為「管理員」，我希望「能查看並管理系統內的所有使用者」，以便「維護系統的安全與秩序」。

## 3. 非功能需求
- **技術限制**：
  - 後端網頁框架：使用 Flask。
  - 模板引擎：使用 Jinja2 渲染前端畫面。
  - 資料庫：使用 SQLite，適合輕量級應用與快速開發。
- **效能與安全考量**：
  - 密碼必須經過經過雜湊（Hash）處理，不可明碼儲存。
  - 避免 SQL Injection，使用 ORM 或參數化查詢。
  - 系統響應時間應在合理範圍內，確保使用者體驗流暢。
  - 需要建立基本的使用者認證（Session / Login）機制，確保一般人只能看到自己的記帳紀錄。

## 4. MVP 範圍 (Minimum Viable Product)

### Must Have (必要功能)
- 使用者註冊與登入/登出。
- 新增、瀏覽、編輯、刪除收支紀錄（CRUD）。
- 提供基本的預設收支分類。
- 權限隔離（每個使用者只能查看與編輯自己的紀錄）。

### Should Have (應有功能)
- 簡易的關鍵字與日期搜尋功能。
- 每月的收支總額簡單統計。
- 管理員後台介面，可查看系統用戶列表。

### Nice to Have (加分功能)
- 視覺化的統計圖表（如圓餅圖顯示各分類花費）。
- 使用者可自訂專屬的收支分類。
- 資料匯出為 CSV 功能。
